# Интеграционные заметки vk_bot_service

Этот файл фиксирует решения и единственную правку вне vk_bot_service.
Сам сервис полностью изолирован (своя БД, свои воркеры). Существующую
логику платформы он не меняет.

## Получение vk_id (через события auth)

Бот держит локальный users_cache (user_id <-> vk_id + роль/имя/студия),
наполняемый событиями auth_events (user.created/updated/deactivated).

vk_id приходит как поле payload этих событий - источник истины auth.
Бот НЕ угадывает связь vk_id<->user_id (это знает только auth, где vk_id
хранится у User). Это исключает риск выдать доступ к чужому аккаунту.

ПРАВКА AUTH #1 (аддитивная): добавить vk_id в payload событий user.*
- Поле vk_id уже есть в МОДЕЛИ User (Блок 1). Добавляем его лишь в
  Pydantic-схему события (UserCreatedPayload / UserUpdatedPayload) как
  Optional[int|str].
- По правилу контракта проекта добавление optional-поля - совместимое
  изменение, schema_version НЕ меняется.
- Consumer'ы (schedule/admin/crm/notification) читают payload по ключам и
  лишнее поле игнорируют - ничего не ломается.

## ПРАВКА AUTH #2 (аддитивная): internal vk-login (нужна для действий преподавателя)

Действия преподавателя в боте (отмена занятия, просмотр расписания)
требуют вызова Schedule Service. Ручки schedule защищены JWT с проверкой
роли и владения (get_current_teacher / check_teacher_access) - то есть
действие должно идти ОТ ИМЕНИ пользователя, а не по internal-ключу.

Бот получает vk_id из доверенного источника (Long Poll сообщества), но
существующий роут POST /api/v1/auth/vk/login требует обмена VK-КОДА
(code/device_id/code_verifier из окна VK ID на фронте) - у бота кода нет.

При этом сервисный метод auth_service.vk_login(vk_id, ...) уже принимает
готовый vk_id и выдаёт пару токенов. Нужна тонкая обёртка-роут, доступная
по X-Internal-API-Key:

    POST /api/v1/auth/internal/vk-login
    headers: X-Internal-API-Key
    body: { "vk_id": <int|str> }
    -> переиспользует auth_service.vk_login(vk_id=...)
    -> возвращает { access_token, refresh_token, token_type } и user

Свойства:
- строго аддитивно: новый роут, существующие не меняются;
- переиспользует готовую логику vk_login (ничего не дублируется);
- доступ только internal (X-Internal-API-Key), наружу не торчит.

Получив токены, бот хранит refresh-токен пользователя в своей БД и
обновляет access через уже готовый POST /api/v1/auth/refresh (он умеет
брать refresh из тела запроса - сделано в Блоке 1 ровно для бота).

Эта правка оформляется отдельным шагом и согласуется до внесения, чтобы
не затронуть работающую авторизацию.
